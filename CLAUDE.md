# CLAUDE.md

Pairfect is an AI-powered food and drink pairing platform for restaurants. Diners set taste preferences (flavor profile, spice level, allergens, diet, alcohol tolerance) and receive personalized menu pairings (dish + drink + optional dessert) generated by an LLM. Restaurant staff manage menus with rich metadata (allergens, diet tags, taste profiles). The UI is in English.

## Tech Stack

| Component | Technology |
|-----------|------------|
| Framework | Next.js 16 (App Router, Turbopack) |
| UI | React 19, shadcn/ui, Base UI, Tailwind CSS 4, Lucide Icons |
| Backend | Convex (serverless, reactive, ACID) |
| Auth | WorkOS AuthKit (JWT/OIDC via AuthKit) |
| AI/LLM | Google Gemini 3 Flash via OpenRouter (OpenAI SDK client) |
| Animations | Motion (Framer Motion) |
| Fonts | Playfair Display (headings), Plus Jakarta Sans (body) |
| Linting | ESLint + Prettier |
| Deployment | Vercel (web) + Convex Deploy (backend) |

## Data Models

### Core Entities
- **users** — Auth-linked records with `tokenIdentifier`, `email`, `name`, `role` (admin | diner)
- **restaurants** — Restaurant profiles with `name`, `slug` (unique, indexed), `description`, `logoStorageId`
- **restaurantMembers** — Staff roles (owner | staff) linking users to restaurants

### Menu & Allergens
- **menuItems** — Items with `category`, `allergenIds`, `dietTags`, `containsAlcohol`, `alcoholLevel`, `tasteProfile`, `spiceLevel`, `price`, `imageStorageId`
- **allergens** — EU 14 mandatory allergens (Regulation No 1169/2011), seeded and immutable

### Preferences & Recommendations
- **clientProfiles** — Global diner preferences: `tasteProfile`, `spiceLevel`, `allergenIdsToAvoid`, `dietPreference`, `alcoholTolerance`, `sweetTooth`
- **restaurantClientProfiles** — Per-restaurant preference overrides (same fields + `restaurantId`)
- **recommendations** — AI-generated pairings with `items` array (`menuItemId`, `menuItemName`, `matchPercentage`, `reason`) + `preferenceSnapshot`

### Invitations
- **pendingInvitations** — Email-based staff invitations; auto-claimed when invited user signs up

## Key Patterns

**Authentication**: WorkOS OAuth → `/callback` route → JWT extracted via AuthKit → `ConvexProviderWithAuth` sends token → Convex validates via JWKS. `useEnsureUser` hook creates/updates the Convex user record on first login (defaults to "diner" role).

**Preference profiles**: Global preferences in `clientProfiles`, per-restaurant overrides in `restaurantClientProfiles`. Always upsert (insert or update), never separate create/update flows. UI offers "use global" vs "customize" toggle.

**AI pairing generation**: `recommendations.ts` action → fetches menu items + allergens → constructs LLM prompt → calls Gemini via OpenRouter (OpenAI SDK client) → parses JSON response (2-3 pairings) → stores in `recommendations` with preference snapshot.

**Staff invitations**: Admin invites via email → `pendingInvitations` entry. When the invited user signs up, `ensureUser` claims pending invitations and creates `restaurantMembers` records automatically.

**Role-based routing**: Diners see restaurant listing on home page. Staff/admins are redirected to admin dashboard. Permission checks use `requireRestaurantMember()` in Convex functions.

**ID strategy**: Convex auto-generates `_id` (string) for all documents. Always use `v.id("tableName")` for references and `Id<"tableName">` for TypeScript types.

## Convex

This project uses [Convex](https://convex.dev) as its backend.

When working on Convex code, **always read `convex/_generated/ai/guidelines.md` first** for important guidelines on how to correctly use Convex APIs and patterns. The file contains rules that override what you may have learned about Convex from training data.

Convex agent skills for common tasks can be installed by running `npx convex ai-files install`.

## Naming Conventions

### Function prefixes

| Prefix | Returns | Example |
|--------|---------|---------|
| `find` | `T \| null` (nullable) | `findClientByEmail` |
| `get` | `T` guaranteed (throws if not found) | `getRestaurantById` |
| `list` | `T[]` (array) | `listMenuItemsByRestaurant` |
| `create` | New entity ID | `createRestaurant` |
| `update` | Modifies existing entity | `updateMenuItem` |
| `delete` | Removes entity | `deleteMenuItem` |
| `ensure` | Upsert (create or update) | `ensureUser` |
| `require` | Returns T or throws auth error | `requireUser`, `requireRestaurantMember` |
| `current` | Scoped to authenticated user | `currentUser` |

### General

- **Files**: kebab-case (`client-profiles.tsx`). PascalCase only for React component files matching the component name.
- **React components**: Arrow functions (`const MyComponent = () => {}`), never `function` declarations.
- **Convex tables**: camelCase, plural (`menuItems`, `clientProfiles`).
- **Validators**: Defined once in `convex/validators.ts`, reused across schema and functions.
- **Type imports**: Use `Id<"tableName">` and `Doc<"tableName">` from `convex/_generated/dataModel`.

## Rules

- **Styling**: Tailwind CSS 4 utility classes. No inline styles. Dark mode only (class strategy).
- **UI components**: shadcn/ui from `components/ui/`. App-specific components in `components/`.
- **State**: Convex reactive queries for server state. Local React state for forms and ephemeral UI state.
- **Data writes**: Always use Convex mutations (`useMutation`), not Server Actions.
- **Auth checks**: Use `authHelpers.ts` (`getCurrentUser`, `requireUser`, `requireRestaurantMember`) in all Convex functions that need auth.
- **Queries**: Always use `.withIndex()`, never `.filter()`. Use `.take(n)` or `.paginate()`, not unbounded `.collect()`.
- **Loading states**: Handle `undefined` (loading), `null` (no data), and data as three distinct states.
- **Icons**: Lucide React (`lucide-react`).
